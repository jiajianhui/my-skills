# 产品需求拆解

这个 skill 用于把已经相对明确的产品意图，整理成清晰、可评审、可继续落地的需求定义。

它负责把“想做什么”说清楚，但不负责进一步排期或分配工程任务。若当前问题本身还不清楚，先使用 `problem-insight`；若已经进入执行拆解，使用 `task-breakdown`。

## 工作流程

1. 先确认要解决的问题、目标用户和预期结果。
2. 区分业务目标、用户价值和具体功能方案。
3. 明确范围、非范围和版本边界，避免需求膨胀。
4. 梳理核心用户流程、关键状态和边界情况。
5. 识别依赖、风险、限制条件和待确认问题。
6. 将需求整理成可评审、可沟通、可验收的结构。

## 输出要求

输出应包含：

- 问题定义
- 用户价值
- 目标
- 范围
- 非范围
- 用户流程
- 边界情况
- 依赖
- 限制条件
- 风险
- 待确认问题
- 验收标准

## 约束

- 不要掩盖模糊点，要主动暴露出来。
- 需求必须可验证。
- 区分“假设”和“已确认事实”。
- 不要把实现细节误写成需求目标。
- 不要把任务拆解、排期和分工提前写进 PRD 主体。
