# Git 提交规范

这个 skill 用于执行代码提交，目标是让提交过程稳定、可复用、符合固定格式。

## 触发场景

当用户表达以下意图时使用：

- 提交代码
- 帮我提交
- commit 一下
- 按规范提交
- 用固定格式提交

## 默认流程

1. 先检查 `git status --short`，确认工作区范围。
2. 只暂存与当前任务直接相关的文件，不顺手提交无关变更。
3. 如有必要，先查看 `git diff --staged` 或相关文件内容，确认提交内容准确。
4. 再执行 `git commit`。
5. 返回提交信息和 commit hash。

## 提交信息规则

默认使用中文提交信息，简洁、直接，描述本次变更的核心动作。

默认格式：

`<type>: <中文提交说明>; 由AI提交🌟`

示例：

- `docs: 补充关键文件注释和贡献说明; 由AI提交🌟`
- `feat: 新增新闻抓取结果缓存接口; 由AI提交🌟`
- `fix: 修复 RSS 解析异常处理; 由AI提交🌟`
- `refactor: 清理未使用参数并简化解析流程; 由AI提交🌟`

## 类型选择规则

- `feat`：新增功能
- `fix`：修复缺陷
- `docs`：文档、注释、README、AGENTS 等说明性改动
- `refactor`：重构或清理代码，不改变功能
- `style`：纯格式调整，不改逻辑
- `test`：测试相关变更
- `chore`：配置、依赖、脚本或其他杂项维护

## 提交约束

- 不提交 `node_modules/`、IDE 配置、缓存文件或其他本地产物，除非用户明确要求。
- 不默认把所有改动一次性提交；只提交与当前需求相关的内容。
- 如果发现存在无关修改，要主动避开，必要时明确指出。
- 如果提交会遗漏关键文件，先补齐再提交。
- 如果用户明确给了提交内容，优先使用用户提供的描述，并补全前缀类型与尾缀 `; 由AI提交🌟`。
- 如果用户没有指定类型，则根据改动内容自动选择最合适的类型。

## 输出方式

提交完成后，简洁返回：

- 实际提交信息
- commit hash

不要展开冗长总结，除非用户要求。
