# 代码学习与流程讲解

这个 skill 用于把陌生代码讲清楚，重点是帮助用户形成“从输入到输出”的心智模型，而不是做抽象的架构空谈。

## 触发场景

当用户表达以下意图时使用：

- 看不懂某个文件或函数
- 帮我梳理执行流程
- 从原始数据到最终结果经历了哪些步骤
- 生成学习文档、流程文档、Markdown 讲解
- 给代码补适合学习的注释
- 把 XML、JSON、API 响应如何变成 JS 对象讲清楚

## 默认流程

1. 先找入口函数和真实执行顺序。
2. 找出输入、中间形态、输出。
3. 用“原始输入 -> 中间对象 -> 最终结果”梳理数据流。
4. 只解释当前用户真正卡住的语法、API 和关键调用点，不做整份代码逐行翻译。
5. 如需写文档，按步骤输出，并给最小可读示例。
6. 如需补注释，优先写函数职责；只在关键调用点补必要的语法说明。

## 讲解方式

- 优先讲“这一步输入什么，输出什么”。
- 优先讲“它在做什么”，再讲“为什么这么做”。
- 避免空话，例如“进行处理”“完成解析”，要明确处理前后格式。
- 避免术语堆砌、翻译腔和高大上表达。
- 对初学者，必要时明确区分：原始字符串、XML、JSON、JS 对象、最终结果对象。
- 如果用户只卡一行，就先解释那一行，不要强行扩展成整份教程。

## 参考模板

如需直接生成学习文档，可参考 `references/code-flow-template.md`。

## 文档写法

默认按下面结构组织：

1. 一句话主线
2. 对应函数和执行顺序
3. 每一步的输入长什么样
4. 每一步的输出长什么样
5. 为什么需要这一步
6. 最小代码示例或数据示例

推荐主线写法：

`原始输入 -> 解析/转换 -> 中间对象 -> 业务处理 -> 最终输出`

## 注释约束

- 主步骤注释放在关键流程上，按执行顺序编号。
- 语法或 API 解释只写在用户容易卡住的那一行附近。
- 不要把主步骤注释和补充说明混成一大段。
- 注释要简单、清楚、像人话，不要端着。
- 如果某个注释只是重复代码字面意思，就删掉。

## 输出约束

- 默认用中文。
- 先给主线，再给局部解释。
- 生成 Markdown 时尽量简洁，示例要和真实代码一致。
- 目标是帮助用户建立心智模型，而不是展示你知道多少术语。
